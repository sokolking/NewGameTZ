# План: серверная синхронизация и мультиплеер

## 1. Цели

- Все действия пользователя выполняются локально; в **конце хода** данные отправляются на сервер.
- Сервер — **единственный источник истины**: знает состояние боя, позиции всех юнитов, ОД, штрафы.
- Два игрока играют **одновременно**: каждый делает ход у себя локально и по завершении хода отправляет его на сервер.
- Сервер **синхронизирует** ход: когда получил завершение хода от всех участников — разрешает конфликты и отправляет результат всем клиентам.
- Клиенты **визуализируют** полученный результат: каждый видит и свои движения (например, 5 клеток), и движения оппонента (например, 7 клеток) после синхронизации.
- **Обработка ошибок**: конфликты (два юнита на один гекс), сетевые сбои, таймауты, невалидные данные.
- **Бой привязан к серверу**: сервер знает длительность раунда и какой сейчас раунд. В **начале раунда** сервер один раз сообщает клиентам номер раунда и длительность (сек); у клиента запускается **локальный таймер**. Сервер **тоже** считает время раунда; при истечении таймера на сервере раунд принудительно завершается у всех (даже если клиент «взломает» и не завершит свой таймер — сервер разошлёт результат и переведёт всех в следующий раунд).
- **Одновременное выполнение хода**: после получения результата от сервера все юниты двигаются **одновременно** (параллельная анимация). Конфликт «кто первым встал на клетку» решается пошагово: кто в симуляции сервера первым занял клетку — тот её получает; второй остаётся на своей клетке, ОД с него списываются, перехода не происходит.

## 2. Архитектура

```
┌─────────────────┐                    ┌─────────────────┐
│   Клиент A      │                    │   Клиент B       │
│  (Unity)        │                    │   (Unity)        │
│  - локальные    │                    │  - локальные     │
│    ходы, путь   │                    │    ходы, путь    │
│  - по End Turn  │                    │  - по End Turn   │
│    → отправить  │                    │    → отправить   │
│  - по ответу    │                    │  - по ответу     │
│    → анимация   │                    │    → анимация    │
│    всех юнитов  │                    │    всех юнитов   │
└────────┬────────┘                    └────────┬─────────┘
         │ SubmitTurn(N, path, ...)              │ SubmitTurn(N, path, ...)
         │                                      │
         └──────────────────┬───────────────────┘
                            ▼
                  ┌─────────────────┐
                  │     Сервер      │
                  │  - таймер раунда │
                  │    (свой, без    │
                  │    рассылки)     │
                  │  - номер раунда  │
                  │  - приём ходов   │
                  │  - при таймауте   │
                  │    принудительно │
                  │    завершить раунд│
                  │  - рассылка      │
                  │    TurnResult(N) │
                  │  - раз в начале  │
                  │    раунда:       │
                  │    RoundStarted  │
                  └─────────────────┘
```

- **Транспорт**: **WebSocket** для хода и результата раунда; **HTTP** только для join/poll до старта боя (в Unity — `UnityWebRequest` + `ClientWebSocket`).
- **Формат сообщений**: JSON (удобно для отладки и совместимости).

## 3. Модель данных

### 3.1. Идентификаторы

- **BattleId** — уникальный идентификатор боя (комнаты).
- **PlayerId** — идентификатор игрока в бое (например, "P1", "P2" или GUID).
- **RoundIndex** — номер раунда (0, 1, 2, …). Сервер рассылает его один раз в начале раунда (`RoundStarted`) и в `TurnResult`.

### 3.2. Состояние юнита (на сервере и для синхронизации)

Для каждого игрока/юнита:

- `position` — (col, row) текущая клетка.
- `currentAp`, `maxAp`, `penaltyFraction` — ОД и штраф.
- `apSpentThisTurn`, `stepsTakenThisTurn` — для расчёта штрафов и стоимости шагов (сервер может пересчитывать или доверять клиенту с валидацией).

### 3.3. Отправка хода (Client → Server): `SubmitTurn`

```json
{
  "battleId": "...",
  "playerId": "P1",
  "roundIndex": 0,
  "path": [ {"col": 0, "row": 0}, {"col": 1, "row": 0}, {"col": 2, "row": 0} ],
  "apSpentThisTurn": 14,
  "stepsTakenThisTurn": 2
}
```

- `path` — полный путь за ход: старт + все точки по порядку (как сейчас `_turnPath` в `Player`).
- Сервер может сам пересчитать `apSpentThisTurn` и `stepsTakenThisTurn` по `path` и правилу стоимости шагов, чтобы исключить читы.

### 3.4. Ответ сервера (Server → Client): `TurnResult`

```json
{
  "battleId": "...",
  "roundIndex": 0,
  "results": [
    {
      "playerId": "P1",
      "accepted": true,
      "finalPosition": { "col": 2, "row": 0 },
      "actualPath": [ {"col": 0, "row": 0}, {"col": 1, "row": 0}, {"col": 2, "row": 0} ],
      "currentAp": 86,
      "penaltyFraction": 0,
      "apSpentThisTurn": 14,
      "rejectedReason": null
    },
    {
      "playerId": "P2",
      "accepted": false,
      "finalPosition": { "col": 1, "row": 1 },
      "actualPath": [ {"col": 1, "row": 1}, {"col": 2, "row": 0} ],
      "currentAp": 80,
      "penaltyFraction": 0.05,
      "apSpentThisTurn": 20,
      "rejectedReason": "Target hex (2,0) already occupied by P1"
    }
  ]
}
```

- **accepted** — весь заявленный путь выполнен (true) или часть отклонена из‑за конфликта (false).
- **finalPosition** — итоговая позиция по решению сервера.
- **actualPath** — путь, который реально прошёл юнит (может быть короче, если конец обрезан из‑за занятой клетки).
- **rejectedReason** — текст для UI («клетка занята», «невалидный путь» и т.д.).
- ОД и штрафы в ответе — канонические значения с сервера; клиент подставляет их в локальное состояние.

### 3.5. Старт раунда (Server → Client): `RoundStarted` — один раз в начале раунда

Сервер **не** рассылает время с периодичностью (это дорого по трафику и нагрузке). Вместо этого в **начале каждого раунда** сервер один раз отправляет участникам боя сообщение:

```json
{
  "battleId": "...",
  "roundIndex": 3,
  "roundDuration": 100.0
}
```

- **roundIndex** — номер раунда (0, 1, 2, …). Клиент показывает в UI («Ход 4»).
- **roundDuration** — длительность раунда в секундах. Клиент **запускает локальный таймер** от этого значения (обратный отсчёт) и отображает «осталось T сек» в UI.

Сервер **тоже** ведёт свой таймер раунда. Раунд завершается в двух случаях: **(1)** все участники уже отправили ход — сервер **сразу** завершает раунд (не дожидаясь таймера), разрешает конфликты и рассылает `TurnResult`; **(2)** истёк отведённый на ход срок на сервере — сервер принудительно завершает раунд. При таймауте **клиент** отправляет данные **как есть** (текущий прогресс хода: путь на момент истечения времени); если клиент так и не прислал ход (сбой), сервер считает для него пустой путь. Таким образом: пользователи могут закончить ход досрочно; если не успели в срок — отправляется текущий прогресс, сервер завершает ход.

### 3.6. Матчмейкинг: Find Game и старт боя (Server → Client): `BattleStarted`

В главном меню — кнопка **«Find Game»** (вместо «Start Game»). Клиент подключается к серверу и отправляет запрос **«встать в очередь»** (например, `JoinQueue`). Сервер держит очередь; когда в очереди набирается **2 игрока**, создаёт бой и рассылает каждому участнику сообщение:

```json
{
  "battleId": "battle-uuid",
  "playerId": "P1",
  "players": [ {"playerId": "P1", "col": 0, "row": 0}, {"playerId": "P2", "col": 24, "row": 39} ],
  "roundDuration": 100.0
}
```

- **battleId**, **playerId** — клиент сохраняет для всех последующих сообщений (SubmitTurn, RoundStarted, TurnResult).
- **players** — начальные позиции всех участников (col, row); клиент расставляет юниты на карте, себя помечает как локального игрока по `playerId`.
- **roundDuration** — длительность раунда в секундах (для первого и следующих раундов). После загрузки боя сервер пришлёт `RoundStarted(roundIndex=0, roundDuration)` — по нему клиент запускает локальный таймер и начинает приём ходов.

### 3.7. Конфликт: занятая клетка (одновременная пошаговая симуляция)

- Правило: **одна клетка — один юнит**; решаем не «по финальным позициям», а **пошагово**, как будто все идут одновременно.
- На каждом **шаге симуляции** (шаг 1, шаг 2, …) обрабатываем игроков в **порядке отправки хода в этом раунде**: кто раньше отправил `SubmitTurn` (раньше «закончил ход») — тот выше в приоритете. Кто в этом порядке первым претендует на клетку — тот её занимает; второй, если хотел на ту же клетку, **не переходит** — остаётся на своей текущей клетке, ОД за этот шаг списываются.
- Итог: оба визуально «идут одновременно», но при конфликте клетку получает тот, кто раньше завершил/отправил ход. (В будущем приоритет может задаваться параметрами игрока — на данном этапе не реализуем.)

## 4. Логика сервера

### 4.1. Состояние боя на сервере

- Список игроков, их текущие позиции, ОД, штрафы.
- **Текущий раунд**: `roundIndex`, длительность `roundDuration`, **локальный на сервере** таймер раунда (сервер тикает его сам, **никому не рассылает**). В начале раунда сервер один раз рассылает `RoundStarted(roundIndex, roundDuration)` — клиенты по нему запускают свой локальный таймер.
- Очередь/буфер отправленных ходов по `roundIndex`: для каждого раунда — по одному сообщению от каждого игрока. Раунд завершается **как только** выполнилось одно из двух: **(а)** все отправили ход — сервер сразу разрешает ход и рассылает `TurnResult`; **(б)** истёк таймер раунда на сервере — сервер закрывает приём ходов. К этому моменту каждый клиент при истечении своего таймера уже отправил **текущий прогресс** (путь на момент таймаута); кто не прислал (сбой) — считается с пустым путём. Рассылка `TurnResult`, затем `roundIndex + 1`, сброс таймера, рассылка `RoundStarted`.

### 4.2. Алгоритм разрешения хода (одновременная пошаговая симуляция)

1. Дождаться `SubmitTurn(roundIndex=N)` от **всех** игроков или срабатывания таймаута раунда. При таймауте клиенты к этому моменту уже отправили текущий прогресс (путь на момент истечения времени); не приславшие — считаются с пустым путём.
2. Зафиксировать порядок игроков на этот раунд **по порядку поступления SubmitTurn**: кто раньше отправил ход — тот выше в приоритете. Этот порядок определяет «кто первым встал на клетку» на каждом шаге симуляции.
3. **Пошаговая симуляция** (все идут «одновременно»):
   - Текущие позиции всех: `pos[playerId] = path[i][0]` (старт).
   - Множество занятых клеток: изначально все стартовые позиции.
   - `maxSteps = max(length(path_i) - 1)` по всем игрокам.
   - Для `step = 1 .. maxSteps`:
     - Для каждого игрока в порядке отправки хода в этом раунде (кто раньше прислал SubmitTurn — тот раньше в очереди):
       - Целевая клетка = `path[playerId][step]` (если у игрока путь короче — он уже стоит на месте, шаг пропускаем или считаем «остаться»).
       - Если целевая клетка **уже занята** (кем-то из уже обработанных на этом шаге или кем-то на предыдущих шагах): игрок **остаётся** на `pos[playerId]`, но ОД за этот шаг списываются (записываем в actualPath текущую позицию повторно или не добавляем шаг; итоговая позиция не меняется). Помечаем `accepted = false`, записываем причину.
       - Если целевая клетка **свободна**: игрок переходит, `pos[playerId] = target`, добавляем клетку в занятые, в actualPath добавляем target.
   - После цикла для каждого игрока пересчитать ОД и штрафы по фактически выполненным шагам (по `actualPath` и правилам GetStepCost / предпоследний-последний шаг).
4. Сформировать `TurnResult`: для каждого игрока `finalPosition`, `actualPath`, `currentAp`, `penaltyFraction`, `accepted`, `rejectedReason`. Разослать всем клиентам.
5. Обновить состояние боя на сервере (позиции, ОД, штрафы), увеличить `roundIndex`, сбросить таймер раунда и **один раз** разослать `RoundStarted` для нового раунда.

### 4.3. Обработка ошибок на сервере

- **Дубликат хода**: повторный `SubmitTurn` от того же `playerId` для того же `roundIndex` — игнорировать или перезаписать (политика на выбор).
- **Неверный roundIndex**: ожидаем N, пришёл N+1 или N-1 — ответ с ошибкой, не менять состояние.
- **Неизвестный battleId / playerId**: ответ 4xx с кодом и сообщением.
- **Таймаут раунда**: время на ход задано. По истечении таймера **клиент** отправляет текущий прогресс (путь как есть на момент таймаута); сервер завершает раунд и разрешает ход (для не приславших — пустой путь). Досрочное завершение хода пользователем — по нажатию «Завершить ход», данные отправляются сразу.

## 5. Логика клиента (Unity)

### 5.1. Режимы работы

- **Онлайн (PvP и одиночка через сервер)** — `End Turn` всегда идёт через сервер:
  1. **Таймер и раунд**: в начале раунда клиент получает от сервера **один раз** `RoundStarted(roundIndex, roundDuration)`. Запускает **локальный** обратный отсчёт от `roundDuration` и показывает в UI «Ход N», «Осталось T сек». Сервер отдельно считает время; при истечении на сервере раунд завершается у всех (клиент получит `TurnResult`), даже если локальный таймер клиента был «взломан».
  2. При «Завершить ход» (досрочно) или при **истечении локального таймера**: собрать payload `SubmitTurn` с **текущим прогрессом** (path, apSpent, stepsTaken, roundIndex — путь на данный момент, как есть), отправить на сервер. Локально показать «Ожидание других игроков…» (или «Ход отправлен»), не менять позиции до ответа. То есть при таймауте клиент не ждёт ввода — отправляет то, что успел набрать за раунд.
  3. По получении `TurnResult`:
     - Применить к **каждому** юниту (своему и чужим): `finalPosition`, `actualPath`, `currentAp`, `penaltyFraction`, обновить счётчик раунда из сервера.
     - **Одновременная анимация**: запустить движение по `actualPath` для **всех** юнитов **параллельно** (не «сначала A до конца, потом B»). Все идут в один момент времени; кто путь короче — тот раньше останавливается.
     - Обновить UI (ОД, лог). Следующий раунд и таймер — из следующего сообщения `RoundStarted`, после чего снова запустить локальный таймер.

### 5.2. Разделение ответственности

- **Player** — «один юнит»: позиция, ОД, путь, стоимость шагов. Имеет возможность применять «внешнее» состояние с сервера (`ApplyServerTurnResult(finalPosition, actualPath, ap, penalty)`), а не считать исход раунда сам.
- **GameSession / NetworkManager**:
  - Подключение: HTTP (join/poll), затем WebSocket на время боя.
  - Отправка `SubmitTurn` при вызове «завершить ход» из UI.
  - Приём `TurnResult`, парсинг, вызов применения состояния ко всем юнитам и запуск анимаций.
- **Локальный vs удалённый юнит**: на сцене два (или N) объекта «юнит»: один привязан к локальному игроку (управление, подсветка хода), остальные — только визуал + применение `TurnResult`. Либо один компонент `Player` с флагом `isLocal`, второй юнит — `RemotePlayer` или `PlayerView` с данными с сервера.

### 5.3. Обработка ошибок на клиенте

- **Сеть недоступна / таймаут**:
  - При отправке `SubmitTurn`: повторные попытки (с ограничением), затем сообщение «Не удалось отправить ход. Проверьте соединение.» Не увеличивать номер раунда локально до успешной отправки и получения ответа.
- **Ответ сервера с ошибкой** (4xx, или поле `error` в JSON):
  - Показать сообщение пользователю.
  - Не применять невалидное состояние; при необходимости повторно отправить ход или оставить локальное состояние как «ожидание».
- **TurnResult с accepted: false** для своего юнита:
  - Показать уведомление: «Клетка занята, ход частично отменён» (или текст из `rejectedReason`).
  - Применить `finalPosition` и `actualPath` с сервера — юнит визуально остаётся или откатывается на допустимую позицию, ОД обновлены по серверу.
- **Потеря соединения во время ожидания TurnResult**:
  - Таймаут ожидания (например, 30 с); по истечении — «Сервер не ответил. Попробуйте переподключиться.» Опционально: повторная отправка хода или запрос состояния боя (отдельный метод API).

## 6. Этапы реализации

### Этап 1: Инфраструктура и контракты (без реального сервера)

- Ввести **модели данных** в Unity (C# классы / struct для `SubmitTurn`, `TurnResult`, `PlayerTurnResult`).
- Добавить **GameSession** (или **BattleClient**): метод `SubmitTurnLocal(path, apSpent, stepsTaken)` — пока только формирует payload и логирует его (или пишет в файл). Метод `ApplyTurnResult(result)` — заглушка.
- В **ActionPointsUI** / месте вызова «Завершить ход»: в режиме «онлайн» вызывать `GameSession.SubmitTurnLocal(...)` после текущей логики `Player` (путь и ОД уже есть), не меняя пока поведение анимации.
- Цель: подготовить точки входа и форматы данных.

### Этап 2: Сервер (минимальный) и владение раундом

- Поднять простой сервер (Node.js + WebSocket или C# ASP.NET Core + WebSockets / SignalR).
- **Раунд на сервере**: сервер хранит `roundIndex`, `roundDuration` и **свой** таймер раунда (тикает локально, **никому не рассылает**). В начале раунда рассылает участникам **один раз** `RoundStarted(roundIndex, roundDuration)`. При истечении таймера или при получении ходов от всех — завершает раунд, рассылает `TurnResult`, затем `roundIndex++`, сброс таймера и рассылка нового `RoundStarted`.
- Подключение по `battleId`, приём `SubmitTurn(roundIndex, path, …)`. Пока **без пошаговых конфликтов**: каждый игрок «принят», путь как есть. Формировать `TurnResult` с `accepted: true` и теми же путями.
- Рассылка `TurnResult` всем клиентам в комнате; затем обновление раунда на сервере (`roundIndex++`, сброс таймера) и **один раз** рассылка нового `RoundStarted`.
- Клиент: по `RoundStarted` запускает локальный таймер; отправка по WebSocket/HTTP, приём `TurnResult`, вызов `ApplyTurnResult` с логированием.

### Этап 3: Пошаговое одновременное разрешение конфликтов на сервере

- Реализовать алгоритм из п. 4.2: **пошаговая симуляция** — на каждом шаге обрабатывать игроков в **порядке отправки хода** (кто раньше отправил SubmitTurn в этом раунде — тот выше в приоритете); кто первым претендует на клетку, тот её получает; второй остаётся на месте, ОД списываются.
- Пересчёт ОД и штрафов по фактическому `actualPath` (у кого шаг отменён — тот не продвинулся, но стоимость шага всё равно списана).
- В `TurnResult` для игроков с отменёнными шагами: `accepted: false`, `rejectedReason`, `actualPath` (только реально пройденные клетки), `finalPosition`.
- Клиент: при `ApplyTurnResult` использовать только данные с сервера для всех юнитов; анимация — одновременная (этап 4).

### Этап 4: Применение результата на клиенте и одновременная анимация

- Для **каждого** игрока в `results`: найти или создать юнит на сцене (по `playerId`), применить к нему данные с сервера (`finalPosition`, `actualPath`, ОД, штраф). Локальный игрок тоже обновляется из `TurnResult` — единый источник истины.
- **Одновременная анимация**: воспроизвести движение по `actualPath` для **всех** юнитов **в один момент** — либо N корутин (по одной на юнита), стартующих в одном кадре, либо одна корутина, которая каждый кадр двигает все юниты по их путям. В итоге не «сначала A прошёл весь путь, потом B», а оба идут параллельно; кто путь короче — тот раньше останавливается на финальной клетке.
- После окончания анимаций обновить UI (ОД, штраф, лог). Номер раунда и запуск локального таймера — из следующего `RoundStarted`, который сервер пришлёт после перехода в следующий раунд.

### Этап 5: Обработка ошибок

- Клиент: таймауты запроса, повторные попытки, сообщения в UI («Ожидание сервера…», «Клетка занята», «Нет соединения»).
- Сервер: валидация входящих данных, корректные коды и сообщения при неверном `roundIndex`, неизвестном `battleId`/`playerId`, дубликатах.
- **Обязательно**: таймаут раунда на сервере. Когда истёк серверный таймер — раунд принудительно завершается. К этому моменту клиенты при истечении своего таймера уже отправили текущий прогресс хода; у кого нет отправленного хода (сбой) — пустой путь. Рассылка `TurnResult` всем.

### Этап 6: Матчмейкинг и жизненный цикл боя

- **Главное меню:** вместо «Start Game» — кнопка **«Find Game»**. По нажатию клиент подключается к серверу и встаёт в **очередь на поиск боя**.
- **Сервер:** ведёт очередь (очередь на матч). Как только в очереди набирается нужное число игроков (для начала **2**), сервер создаёт бой, назначает `battleId` и `playerId` участникам (например P1, P2), рассылает им сообщение **BattleStarted** (battleId, playerId, начальные позиции, список игроков) и переводит подключения в контекст этого боя.
- **Клиент:** по получении **BattleStarted** загружает игровую сцену (или переходит в режим боя), подставляет своего юнита и оппонента по данным с сервера, ждёт **RoundStarted(0, roundDuration)** и начинает первый раунд (локальный таймер, приём SubmitTurn).
- Старт боя: начальные позиции задаёт сервер (например, фиксированные для P1 и P2). Сервер рассылает **один раз** `RoundStarted(roundIndex=0, roundDuration)`; клиенты запускают локальный таймер и принимают `SubmitTurn` по `roundIndex`.
